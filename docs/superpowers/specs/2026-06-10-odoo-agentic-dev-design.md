# Design: @basaltbytes/odoo-agentic-dev v1

Date: 2026-06-10
Status: Approved design (brainstorm output)
Source requirements: [`technical-prd.md`](../../../technical-prd.md)

This document records the decisions that turn the PRD into an implementable design. It does not restate the PRD; where the PRD is already precise (command options, acceptance criteria, safety rules, env table), the PRD is the reference.

## Resolved Decisions

User-confirmed:

1. **Effect v4 beta.** Dependencies: `effect@4.0.0-beta.78` and `@effect/platform-node@4.0.0-beta.78`, pinned exact while beta. In v4 the CLI lives at `effect/unstable/cli` and command execution at `effect/unstable/process`; `@effect/cli` and `@effect/platform` v3 packages are not used. The PRD's package list reflected the v3 layout; the v4 mandate wins.
2. **Scope: full v1.** All eight commands (`info`, `setup`, `up`, `down`, `reset-db`, `update`, `test`, `link-source`), built in the PRD's slice order so the pure core and `info --json` land first.
3. **Flat single package.** The package lives at the repo root (`src/`, `dist/`, `package.json`), not under `packages/`. The existing root `package.json` is extended, not moved.
4. **Flags-only destructive confirmation.** No interactive prompts in v1. Guarded commands refuse and print the exact flag to re-run with.

PRD open questions resolved by default (user approved):

| # | Question | Decision |
| --- | --- | --- |
| 1 | Generated vs supplied compose | Generated by default; `compose.file` recipe field as escape hatch |
| 2 | Recipe formats | TS/JS only: `.ts`, `.mts`, `.js`, `.mjs` |

Registry facts verified 2026-06-10: `oxfmt@0.54.0` exists under that name and binary; `oxlint@1.69.0`; `jiti@2.7.0`.

## Architecture: Pure Core + Effect Shell

Everything that can be pure is pure. The `core/` modules are plain functions over plain data: they take a normalized recipe, Git state, and an env map, and return values or typed errors. They never touch the filesystem, Git, Docker, or `process.env` directly.

Effect services wrap only the genuinely effectful edges in `platform/` and `config/`. Commands compose: read inputs through services, decide through core, act through services.

```text
src/
  cli.ts                     # bin entry: effect/unstable/cli wiring, error rendering, exit codes
  index.ts                   # public API: defineOdooAgenticDevConfig + config types
  commands/                  # one file per CLI command, thin: parse flags -> core -> services
    info.ts setup.ts up.ts down.ts reset-db.ts update.ts test.ts link-source.ts
  core/                      # pure functions, no IO
    worktree-context.ts      # assemble WorktreeContext from recipe + git state + env
    database-name.ts         # naming, sanitization, truncation+hash, shared-branch logic
    port-allocator.ts        # deterministic offset, per-service ports
    compose-project.ts       # compose project name derivation
    compose-model.ts         # generated compose file as data + YAML serialization
    project-recipe.ts        # normalized recipe types
    safety.ts                # shared-db guard, unsafe-name checks, destructive preconditions
    command-plan.ts          # argv/env construction for docker/odoo invocations (pure)
  platform/                  # Effect services (interfaces + live layers)
    git.ts                   # branch/worktree/detached state via git CLI
    docker-compose.ts        # compose up/down/build/exec/run/logs, db readiness wait
    process-supervisor.ts    # companion app lifecycle, prefixed log streaming, Ctrl-C teardown
    odoo-shell.ts            # odoo shell script execution inside the container
    file-system.ts           # thin re-export of platform FileSystem for link-source etc.
  config/
    schema.ts                # Effect Schema for OdooAgenticDevConfigInput, normalization defaults
    load-recipe.ts           # discovery (cwd upward) + jiti load + validation
  errors/
    errors.ts                # tagged error classes per PRD union + render() for CLI output
  testing/
    fake-adapters.ts         # FakeGit, RecordingExecutor, in-memory FS layers
    fixtures.ts              # sample recipes and git states
test/                        # Vitest, mirrors src/ structure
```

Departure from the PRD file map: `errors/` is one module (the classes are small and share a renderer), and `core/` gains `compose-model.ts`, `safety.ts`, `command-plan.ts` — the PRD's `project-layout.ts` responsibilities live in `project-recipe.ts`. Source filenames stay kebab-case.

## Derivation Rules (exact)

These rules are the heart of the tool; they are specified here so tests can be written first.

**Worktree name.** `ODOO_WORKTREE_NAME` env wins. Otherwise the Git branch name. Detached HEAD or not a repo: `<root-dir-basename>-<8-char sha256 of absolute root path>`.

**Database name.**

1. Env override: if `ODOO_DATABASE` and/or `E2E_ODOO_DB` is set, use it; if both set and different, fail with `ConfigValidationError`.
2. Shared branch: if the branch is in `project.sharedBranches`, use `project.sharedDatabase`.
3. Otherwise derive: take the branch name, drop a leading branch-type segment if present (`feature|feat|bugfix|bug|hotfix|fix|chore|task`), sanitize (lowercase; every run of non-`[a-z0-9]` becomes `_`; trim `_`), then prepend `{dbPrefix}_` unless the sanitized name already starts with it. `feature/KL-123-payment-flow` with prefix `kl` → `kl_123_payment_flow` (matches the PRD example).
4. Length: max 63 chars (PostgreSQL limit). If longer, truncate to 54 chars and append `_` + first 8 hex chars of sha256 of the pre-truncation name.
5. Validity gate: final name must match `^[a-z][a-z0-9_]*$`; otherwise `UnsafeDatabaseNameError`.

**Port allocation.** `offset = fnv1a32(databaseName) % ports.range` (defaults: `odooBase` 18069, `companionBase` 28000, `range` 1000). Odoo port = `odooBase + offset`; companion app *i* (in declared order) = `companionBase + offset + i`. Same seed as the database name, so a worktree's URLs are stable and shared-branch worktrees coherently share ports. Explicit `ODOO_HTTP_PORT` env overrides the Odoo port.

**Compose project name.** `sanitize(project.id) + "_" + databaseName`, sanitized to `[a-z0-9_]`. `kriss-laure` + `kl_123_payment_flow` → `kriss_laure_kl_123_payment_flow` (matches the PRD example). Never empty: guarded by `safety.ts`.

**Injected env.** Exactly the PRD table (`ODOO_DATABASE`, `E2E_ODOO_DB`, `ODOO_BASE_URL`, `ODOO_HTTP_PORT`, `ODOO_COMPOSE_PROJECT_NAME`), plus recipe `envAliases` (map of alias name → one of the canonical variables, e.g. `KL_WORKTREE_DB_NAME: "ODOO_DATABASE"`), plus companion-app `env` values with `$VAR` substitution resolved against the canonical set.

## Config

**Public API** (from package root, per PRD): `defineOdooAgenticDevConfig`, `OdooAgenticDevConfigInput`, `OdooAgenticDevConfig`, `OdooProjectConfig`, `OdooRuntimeConfig`, `OdooDatabaseConfig`, `OdooAddonMount`, `CompanionAppConfig`, `PostInitHook`. `defineOdooAgenticDevConfig` is an identity function that preserves literal types; validation happens in the loader.

**Schema/normalization** (Effect Schema): required `project.id`, `project.dbPrefix`, `odoo.version`, `odoo.addons` (non-empty). Defaults: `ports` as above; `odoo.serviceName` "odoo", `odoo.databaseServiceName` "db"; `odoo.postgresImage` "postgres:16"; `database.withoutDemo` "all"; `sharedBranches` `["main", "master"]` when `sharedDatabase` is set. New optional fields beyond the PRD examples: `compose.file` (escape hatch), `envAliases`, `test.profiles` (name → array of CLI-equivalent test options), `odoo.source` (resolver: absolute/relative path, `"sibling"`, or `"docker-only"`).

**Validation failures** (fail closed, per PRD list): duplicate container mount paths, invalid `dbPrefix` (`^[a-z][a-z0-9]*$`), invalid port ranges, unsupported hook types, unsafe companion app names (`^[a-z][a-z0-9-]*$`), addon host paths escaping the repo root unless `allowOutsideRepo: true` on the mount.

**Loading.** `--config <path>` or `ODOO_WORKTREE_CONFIG` env, else discovery: from cwd upward, first directory containing `odoo-agentic-dev.config.{ts,mts,js,mjs}` (that directory becomes the project root). Loaded via `jiti` (internal dependency, interop default export), then schema-validated and normalized with `effect/Schema` (verified present as a stable top-level module in 4.0.0-beta.78).

## Generated Compose

`compose-model.ts` builds a plain object model and serializes it to YAML deterministically (stable key order; serializer is hand-written for this fixed shape — no YAML dependency). The model: a `db` service (postgres image, healthcheck via `pg_isready`, named volume `db-data`, `POSTGRES_USER/PASSWORD/DB` env), an `odoo` service (built from `odoo.dockerfile` + `odoo.imageName` when present, else image `odoo:{version}`; depends_on db healthy; port mapping `{odooHttpPort}:8069`; addon mounts read-only-false; `odoo.configFile` mounted to `/etc/odoo/odoo.conf` when set; named volume `web-data` for filestore). Compose project namespacing isolates volumes per worktree. The file is written to `.odoo-agentic-dev/compose.generated.yml` under the project root (gitignored). With `compose.file` set, generation is skipped and the supplied file is used as-is.

## Database Lifecycle

All Odoo/Postgres work runs through the compose services — no host-side `psql` or Odoo install needed. The argv for each step is constructed by pure `command-plan.ts` functions (unit-testable) and executed by the compose service:

- **reset**: terminate sessions (`psql` in `db` service: `pg_terminate_backend`), `DROP DATABASE IF EXISTS`, `CREATE DATABASE`; remove filestore via `rm -rf /var/lib/odoo/filestore/<db>` in a one-off `odoo` container run.
- **init**: `odoo -d <db> -i <modules> --without-demo=<mode> --stop-after-init`.
- **update**: `odoo -d <db> -u <modules> --stop-after-init`, restarting the long-running Odoo service afterwards unless `--no-restart`.
- **test**: `odoo -d <db> --test-enable [--test-tags ...] --stop-after-init`, exit code propagated; `test.profiles` expand to predefined option sets.
- **hooks**: `odoo-shell-file` / `odoo-shell-inline` pipe code into `odoo shell -d <db>`; `set-ir-config-parameter` is sugar generating a known shell snippet; `command` hooks run on the host with the injected env. Hooks run in declared order.

Safety (per PRD): every destructive path goes through `safety.ts` first — resolved context printed, shared database requires `--allow-shared`, unsafe names and empty compose project names abort.

## Companion Apps & Supervision

`process-supervisor.ts` (Effect scoped resources): starts Odoo first (unless `--odoo-only` skips companions entirely), spawns each companion with cwd/command/args/env from the recipe plus injected context env and the allocated port in `portEnv`. Log lines are prefixed `[pwa]`-style and streamed. On Ctrl-C/SIGTERM, children terminate via Effect scope teardown; the first process to fail is reported by name with its exit code (`CompanionProcessError`).

## Errors

One `Data.TaggedError` class per PRD group, all in `errors/errors.ts`, with a `render(error)` function producing the PRD's required fields: what failed, command attempted (when safe), cwd, exit code, next action. Long child output is truncated to a tail; full logs go to `.odoo-agentic-dev/logs/`. `cli.ts` maps rendered errors to stderr and a non-zero exit code; unknown defects print a bug-report hint.

## Testing Strategy

TDD throughout. Vitest; tests that exercise Effect code run it with `Effect.runPromise` directly — no extra test-framework dependency.

- **Unit (pure core, no Effect needed)**: all derivation rules above including PRD example parity, truncation+hash, env precedence and the `ODOO_DATABASE`/`E2E_ODOO_DB` disagreement, shared-branch handling, port stability, compose model snapshot, command-plan argv construction, safety guards, hook expansion, config normalization/validation cases.
- **Adapter (fakes)**: FakeGit (branch/detached/non-repo), RecordingExecutor asserting exact argv + env maps (e.g. compose up is `docker compose -p <project> -f <file> up -d --build odoo`, never a shell string), in-memory FS for config discovery and link-source.
- **Integration**: `info --json` through the built CLI binary on a fixture project; dry-run `setup` (RecordingExecutor) command-sequence snapshot; generated compose parses as YAML and matches the compose schema shape.
- **Docker integration** (Linux CI, out of default test run): script bringing up the stack on a minimal fixture, asserting db creation and teardown isolation.

## Packaging

ESM-only. `tsc` build to `dist/` with declarations. `package.json`: `bin` { `odoo-agentic-dev`, `oad` } → `dist/cli.js` (shebang), `exports` { ".": dist/index.js + types }, `engines.node >= 20`, pnpm 11 via existing `devEngines`. Scripts exactly as the PRD's baseline (`build`, `lint` oxlint, `format`/`format:check` oxfmt, `test` vitest, `typecheck`). Repo gets git init, `.gitignore` (node_modules, dist, .odoo-agentic-dev), oxlint/oxfmt configs.

## Build Order (implementation milestones)

1. **Foundations + first slice**: package setup, tooling, errors, config schema/loader, pure core (naming/ports/context/safety), `info` with `--json`/`--env`/`--config`. Full unit coverage. This is the PRD's "Suggested First Slice" and proves the v4 wiring.
2. **Compose runtime**: compose model + generation, docker-compose service, `up`/`down` (incl. `--volumes` guard, `--odoo-only`, `--detach`).
3. **Database lifecycle**: command-plan, odoo-shell service, `setup`, `reset-db`, `update`, `test` (+ profiles), hooks.
4. **Companions + source**: process supervisor, companion apps in `up`, `link-source`.
5. **Polish**: README (macOS/Linux/WSL2 docs, pnpm-only), Docker integration script, packaging checks against PRD acceptance criteria.

## Out of Scope (v1, per PRD non-goals)

Production deployment, module scaffolding/migrations, native Windows shells, mandatory `.odoo` symlink, npm/yarn documentation, remote Docker context management, interactive prompts.
